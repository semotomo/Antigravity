## 1. ツールの使い分け

| ツール | 得意なこと | 本プロジェクトでの主な用途 |
|--------|-----------|--------------------------|
| **Antigravity** | 長文コンテキスト維持、複数ファイル横断、ブラウザ操作、並列タスク実行 | 大規模リファクタリング、新機能実装、ドキュメント生成、Webスクレイピング |
| **Codex** | シンプルなコード生成、説明・要約、クイックなバグ修正 | 短いスクリプト生成、コードレビュー補助、アイデア出し |
| ~~**Claude Code**~~ | ~~コード補完、インタラクティブ編集、Serena連携、ターミナル実行~~ | ⚠️ **現在利用停止中**（復帰後：対話的な小幅修正、Serenaメモリ活用） |

### 使い分けの判断基準（現在の2ツール運用）

```
複雑なタスク（複数ファイル、長文コンテキスト必要）
  → Antigravity  ← メイン

単純なコード片、クイックチェック、アイデア出し
  → Codex

※ Claude Code は現在利用停止中。復帰後は対話的作業・Serena連携に活用予定
```

### クロスモデルワークフロー（現在の2ツール運用）

異なるAIを連携させることで相互チェックが可能：

1. **実装**: Antigravity で実装
2. **レビュー**: Codex に「このコードの問題点を挙げて」と依頼
3. **修正**: 指摘を元に Antigravity で修正

## 2. 共通ワークフロー

すべての主要タスクは以下のパターンで進めること：

```
Research（調査） → Plan（計画） → Execute（実装） → Review（レビュー） → Ship（完了）
```

### 各フェーズの詳細

#### 🔍 Research（調査）
- Serena MCP で過去の実装履歴・メモリを必ず確認する
- `key_tasks` を読んで **すでに実装済みの機能を再実装しない**
- Context7 MCP でライブラリの最新ドキュメントを参照する

#### 📋 Plan（計画）
- 変更内容・影響範囲を先に説明し、ユーザーの **y/n 確認** を得てから実装開始
- `docs/<トピック>/implementation_plan.md` に計画を記録する
- 200行を超える大きな変更は必ずステップ分割する

#### ⚒️ Execute（実装）
- 1タスク＝1つのコンテキスト（セッション）に収める（現在は主に Antigravity）
- 実装中に問題が発生した場合は独自判断で代替案を実行せず、ユーザーに相談する
- コメントはすべて **日本語** で記述する

#### 🔎 Review（レビュー）
- `streamlit run app.py` または `python flet_app/main.py` で動作確認
- シフト生成・CSV出力・Supabase連携など主要機能を必ずテストする
- `docs/<トピック>/walkthrough.md` に検証結果を記録する

## 3. CLAUDE.md / AGENTS.md の管理

### 現状

| ファイル | 場所 | 役割 |
|---------|------|------|
| `CLAUDE.md` | プロジェクトルート | Antigravity / Claude Code（復帰後）共通ルール |
| `.agents/AGENTS.md` | `.agents/` | エージェント向け詳細指示（Antigravity が主に参照） |

### ベストプラクティス

**CLAUDE.md に書くべきこと：**
- 言語設定（日本語）
- セッション開始・終了の必須手順
- MCP設定（Serena・Context7）
- ライブラリバージョンチェックルール

**CLAUDE.md に書くべきでないこと（肥大化防止）：**
- 実装の詳細ロジック → Serena メモリに
- 過去のバグ修正履歴 → `implementation_history` メモリに
- タスクのTODOリスト → `key_tasks` メモリに

**更新タイミング：**
- 新しいMCPサーバーを追加したとき
- プロジェクトの構成が大きく変わったとき
- ルールを追加・変更したとき（最低でも月1回見直す）

> [!IMPORTANT]
> CLAUDE.md は **200行以内** を目安に保つこと。長すぎるとAIが読み飛ばす可能性が高まる。
> 重要度が高いルールには `<important>` タグや大文字を活用する。

## 4. Serena MCP によるコンテキスト管理

### メモリ構成（優先度順）

| メモリ名 | 内容 | 更新頻度 |
|---------|------|---------|
| `key_tasks` | 実装済み機能・TODO | セッションごと |
| `project_overview` | プロジェクト基本情報 | 大きな変更時 |
| `code_layout` | ファイル構成・主要関数 | ファイル追加・削除時 |
| `cautions` | 作業時の重要注意事項 | 問題発生時 |
| `implementation_history` | 実装・修正の詳細履歴 | セッションごと |

### セッション開始の必須手順（全AIツール共通）

```
1. list_memories でメモリ一覧を取得
2. key_tasks を読む（再実装防止）
3. 関連するメモリを read_memory で読み込む
4. 作業開始
```

### セッション終了の必須手順（全AIツール共通）

```
1. 実装内容を implementation_history に追記
2. key_tasks の完了チェックを更新
3. logs/<日付>-session.md にセッションサマリーを保存
```

## 5. スキル・コマンド・サブエージェント

### 現在のスキル構成

```
.agents/skills/          ← Antigravity 用スキル
.claude/skills/          ← Claude Code 用スキル（未使用なら今後整理）
```

### スキルの追加方針

1. 繰り返し使う手順はスキル化する（例：Supabaseへのデータアップロード）
2. スキル名は動詞+対象で命名（例：`upload-to-supabase`、`generate-shift`）
3. 公式スキルを参考にしつつ、プロジェクト固有の手順を上書き定義する

### コマンド vs スキル vs サブエージェントの選択

| 種別 | 使いどき |
|------|---------|
| **スラッシュコマンド** | 1つの明確な手順（例：`/sync-supabase`） |
| **スキル** | コンテキストを引き継いで複数ステップで進む処理 |
| **サブエージェント** | 独立して動けるタスク（例：データ収集→整形→アップロード） |

### 推奨スキル・コマンドのアイデア

```
/session-start    — Serenaメモリ読込+タスク確認を自動化
/session-end      — メモリ更新+セッションログ保存を自動化
/check-libs       — ライブラリバージョン確認
/test-app         — アプリ起動+主要機能チェック
```

## 6. プロンプト設計のコツ

### 効果的なプロンプトの書き方

```
## 悪い例
「シフトのバグを直して」

## 良い例
「app.py の generate_shift() 関数で、
 スタッフが5人のとき IndexError が出ます。
 エラートレースバック: [...]
 修正前に当該箇所のコードを確認し、変更内容を説明してから実装してください」
```

### コンテキスト効率化のコツ

- **`/compact`** を活用してコンテキスト長を節約する
- セッションが長くなったら **`/clear`** で整理し、Serenaメモリに引き継ぐ
- 大きな変更は **複数セッションに分割** してコンテキスト枯渇を防ぐ
- 参照させたいファイルは `@ファイル名` で明示的に指定する

### Antigravity 固有のコツ

- 並列的なタスク（例：複数ファイルの同時修正）はAntigravityが得意
- ブラウザ操作が必要なタスク（POSシステム、Supabaseダッシュボード確認）はAntigravityを使う

## 7. Git / バックアップ戦略

### 現状の問題

プロジェクトルートに `app_backup_*.py` などのバックアップファイルが散在している。  
これはAIが誤って古いコードを参照するリスクがある。

### 推奨戦略

```bash
# 機能ごとにコミットを分ける（Squash Mergeを活用）
git add -p           # 変更を選択的にステージング
git commit -m "feat: シフト生成のパフォーマンス改善"

# バックアップファイルは削除してGitに任せる
# 古いコードは git log / git show で参照できる
```

**バックアップファイル（`*_backup_*.py`）の整理計画：**
1. Git履歴が整備されていることを確認
2. バックアップファイルを `.gitignore` に追加
3. 手動バックアップファイルを削除

### .gitignore に追加すべき項目

```gitignore
# バックアップファイル
*_backup_*.py
temp_*.log
temp_*.txt

# ビルド・キャッシュ
__pycache__/
*.pyc
venv/
```

## 8. デバッグ戦略

### 問題発生時のフロー

```
1. エラーメッセージを全文コピー
2. Serena の cautions メモリを確認（既知の問題か？）
3. AIに「エラーの原因の仮説を3つ挙げて」と依頼
4. 1つずつ検証する（全部同時に実装しない）
5. 解決したら cautions に記録する
```

### Codex との連携デバッグ

- **Antigravity/Claude Code** で実装後、**Codex** に「このコードに潜在的なバグはないか」とレビューを依頼する
- クロスモデルレビューにより見落としを減らす

### よくある落とし穴（本プロジェクト固有）

| 問題 | 原因 | 対策 |
|------|------|------|
| `applymap` エラー | pandas 新版は `map` を使う | AGENTS.md に明記済み |
| Flet イベントハンドラの変更 | バージョンアップに伴う破壊的変更 | Context7 で最新ドキュメント確認 |
| Supabase 接続エラー | `secrets.toml` の設定不備 | `secrets.toml.template` を参照 |
| POSシステムのform取得失敗 | セッション管理の問題 | ブラウザ操作はAntigravityで対応 |

## 9. 禁止事項・落とし穴

### 絶対にやってはいけないこと

1. **`shift_settings.json` の手動削除** → ユーザーデータが消失する
2. **`key_tasks` 未確認での実装** → 重複実装が発生する
3. **ユーザー確認なしの実装開始** → ルール違反
4. **問題発生時に独自判断で代替案実行** → 必ずユーザーに相談する
5. **疑わしいパッケージのインストール** → セキュリティリスク（Shai-Hulud攻撃）

### AIへの指示で避けるべきこと

- **曖昧な依頼**（「いい感じにして」）→ 具体的に指示する
- **一度に大量の変更**（「全部リファクタしてください」）→ 小さく分割する
- **検証省略**（「たぶん動いてるはず」）→ 必ず `streamlit run app.py` で確認

## 10. 日常運用チェックリスト

### セッション開始時 ✅

- [ ] Serena メモリを読み込む（特に `key_tasks`）
- [ ] 前回セッションのログを確認する
- [ ] `.library-check-log.json` を確認（7日以上経過なら更新）

### 実装前 ✅

- [ ] 作業計画をユーザーに説明し、「y」を得る
- [ ] 影響を受けるファイルを特定する
- [ ] バックアップが Git にコミット済みか確認する

### 実装後 ✅

- [ ] アプリを起動して動作確認する
- [ ] 主要機能（シフト生成・CSV出力・Supabase連携）をテストする
- [ ] コードコメントが日本語か確認する

### セッション終了時 ✅

- [ ] Serena メモリを更新する（`key_tasks`・`implementation_history`）
- [ ] セッションサマリーを `logs/<日付>-session.md` に保存する
- [ ] 不要な一時ファイル（`temp_*.log` 等）を削除する
