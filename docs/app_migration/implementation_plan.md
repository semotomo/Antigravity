# 実装計画: Fletベース社内ダッシュボード統合ロードマップ

本計画は、既存の Streamlit 業務アプリを Flet ベースの単一アプリへ集約しつつ、将来的に「顧客」「予約」「売上」「在庫」「見積書」「請求書」までを一体化した社内ダッシュボードへ発展させるための実装方針を整理したものです。

完成イメージの要約は [completion_definition.md](/Users/kirik/Desktop/Antigravity/docs/app_migration/completion_definition.md) を参照。

## 現状整理

2026-03-27 時点で、`flet_app/` 側には以下のモジュールがすでに存在します。

- `/dashboard`: 売上サマリー表示
- `/inventory`: 商品移動入力、移動履歴、商品マスタ
- `/shift`: シフト設定、スタッフ設定、シフト生成、履歴
- `/customer-orders`: 客注管理
- `/login`: Supabase 認証

つまり、移行は「これから開始」ではなく、すでに Phase 5 相当の実装が進んでおり、今後は以下 2 つを並行して進める段階です。

1. 既存 Flet モジュールの完成度を上げる
2. 将来の社内統合ダッシュボードを見据えて情報設計とデータ設計を先に揃える

## 目標

1. 売上、在庫、シフト、客注、顧客、予約、見積、請求を 1 つの Flet アプリに統合する
2. Supabase を共通認証・共通データ基盤として利用する
3. 画面単位ではなく「顧客起点」「案件起点」で横断的に情報を追える構成にする
4. 現場運用に耐えるよう、履歴、ステータス、権限、エラーハンドリングを初期段階から揃える

## 開発運用の前提

現在のダッシュボード開発は、ローカルだけで完結させるよりも、以下のループで進める前提とする。

1. ローカルで Flet 画面を修正
2. `git` へ反映
3. Render 上でデプロイ済みダッシュボードを確認
4. 実画面の見え方や動線を見ながら微調整

このため、UI 改修は「コード上は正しい」だけでなく、Render 上での表示確認までを 1 セットとして扱う。

## 将来像

最終的には、単なる POS ダッシュボードではなく、以下の業務をまとめて扱える社内オペレーション基盤を目指します。

- 顧客管理: 顧客台帳、連絡先、来店履歴、対応メモ
- 予約管理: 予約登録、担当者割当、来店ステータス、予約変更
- 売上管理: 店舗別 / 商品別 / 期間別の集計、日次速報、月次確認
- 在庫管理: 商品マスタ、店舗間移動、在庫数確認、欠品アラート
- 客注管理: 取り寄せ、入荷待ち、引渡し、キャンセル管理
- 見積書作成: 顧客・商品・明細からの見積作成、PDF 出力、ステータス管理
- 請求書作成: 見積や売上からの請求書発行、入金確認、未回収管理

## フェーズ計画

### Phase 5: 既存移行機能の安定化とダッシュボード土台の整備

目的:
既存の `dashboard / inventory / shift / customer-orders` を「単独画面の寄せ集め」から「統合ダッシュボードの基盤」へ進化させる。

実装項目:

- `flet_app/main.py`
  - ルーティングを今後の拡張前提で整理
  - 未実装モジュールを追加しやすい構造にする
- `flet_app/components/navigation.py`
  - 今後のモジュール追加を見越したナビゲーション設計へ寄せる
  - 必要に応じて bottom navigation から sidebar 併用へ拡張できるようにする
- `flet_app/pages/dashboard.py`
  - 売上集計だけでなく、各業務モジュールへの入口となるホーム画面へ強化
  - KPI カード、要対応リスト、ショートカットを配置
- `flet_app/pages/shift.py`
  - 既存実装の動作確認と UX 改善
  - 保存、履歴、生成中の遷移制御を継続改善
- `flet_app/pages/inventory.py`
  - 商品移動に加えて、今後の在庫一覧と欠品確認に広げやすい構成へ整理
- `flet_app/pages/customer_orders.py`
  - 顧客マスタ導入前でも運用できるようにしつつ、将来の顧客管理へ接続しやすくする

完了条件:

- 既存 4 モジュール間の導線が明確
- ダッシュボードが単なる売上画面ではなく、業務トップ画面として機能する
- 各モジュールの状態表示、エラー表示、履歴表示の最低限が揃う
- `git` 更新から Render 確認までの調整ループで破綻なく改善できる

### Phase 6: 顧客管理・予約管理の追加

目的:
今後の見積、請求、客注、売上分析の中心になる「顧客」を先に共通化する。

新規候補:

- `flet_app/core/customers/`
- `flet_app/core/reservations/`
- `flet_app/pages/customers.py`
- `flet_app/pages/reservations.py`

実装項目:

- 顧客台帳の作成
- 顧客詳細画面の作成
- 予約一覧、予約登録、予約変更、キャンセル
- 顧客と予約の関連付け
- 将来の売上、見積、請求、客注との横断表示の土台作り

完了条件:

- 顧客をキーに予約と客注をひも付けられる
- 顧客詳細から関連履歴へ遷移できる

### Phase 7: 売上管理・在庫管理の拡張

目的:
既存の「売上ダッシュボード」と「商品移動管理」を、経営確認と現場運用の両方に使えるレベルまで広げる。

新規候補:

- `flet_app/core/sales/`
- `flet_app/pages/sales.py`
- `flet_app/pages/stock_overview.py` または `inventory.py` 内タブ追加

実装項目:

- 日次 / 月次 / 店舗別 / 商品別の売上分析
- 予約由来売上、客注由来売上の識別
- 店舗別在庫一覧
- 閾値ベースの在庫アラート
- 商品移動履歴と現時点在庫の両方を確認できる UI

完了条件:

- ダッシュボード上で売上と在庫の異常が把握できる
- 在庫画面が「移動入力専用」ではなく「在庫管理画面」として機能する

### Phase 8: 見積書・請求書ワークフローの追加

目的:
社内ダッシュボードを業務記録だけでなく、実務の文書発行まで担う基盤にする。

新規候補:

- `flet_app/core/quotes/`
- `flet_app/core/invoices/`
- `flet_app/pages/quotes.py`
- `flet_app/pages/invoices.py`

実装項目:

- 顧客情報から見積作成
- 明細入力、金額計算、消費税計算
- 見積書 PDF 出力
- 請求書発行、請求ステータス管理
- 売上、予約、客注との関連付け

完了条件:

- 顧客起点で見積と請求を追える
- 発行済み文書の再表示とステータス更新が可能

### Phase 9: 横断検索・権限管理・運用機能の整備

目的:
複数モジュールを実運用するために必要な横断機能を追加する。

実装項目:

- ロール別権限制御
- 監査ログ
- 横断検索
- 通知、期限アラート、未対応一覧
- ダッシュボード上のアクション集約

完了条件:

- 担当者ごとに必要な情報へ安全にアクセスできる
- 日々の業務確認がダッシュボード起点で完結する

## 共通データ設計の方針

機能を増やす前に、以下の共通マスタと共通トランザクションを意識して設計する。

共通マスタ:

- 店舗
- スタッフ
- 顧客
- 商品
- 権限ロール

主なトランザクション:

- 予約
- 客注
- 売上
- 在庫移動
- 見積
- 請求

関連の考え方:

- 顧客は予約、客注、見積、請求の中心キーにする
- 商品は売上、在庫、見積明細、請求明細で再利用する
- 店舗は売上、在庫、シフト、予約の共通軸にする
- ステータスはモジュールごとにバラバラにせず、列挙値と表示文言を整理して共通化する

## 直近で優先する実装

次に進める優先順位は以下です。

1. ダッシュボードをホーム画面として再設計する
2. 既存の `inventory / shift / customer-orders` を統合前提の UI に整える
3. 顧客管理の土台を先に追加する
4. 予約管理を追加して顧客軸の業務フローを作る
5. その後に見積書・請求書・高度な売上分析へ進む

## User Review Required

> [!IMPORTANT]
> 将来的に機能が増えるため、ナビゲーションは現在のシンプルな bottom navigation だけでは足りなくなる可能性があります。
> 将来フェーズでは「ダッシュボード + モジュール一覧 + 顧客/案件詳細へ入る導線」を前提に、sidebar または複合ナビゲーションへ変更する想定です。
> また、見積書・請求書を扱う場合は PDF 出力、採番ルール、ステータス管理、権限管理の仕様を早めに確定した方が実装が安定します。

## 検証プラン

### Validation

- Flet アプリ起動後、各モジュールに正常遷移できるか
- ダッシュボードから主要機能へ迷わず移動できるか
- 顧客、予約、売上、在庫、見積、請求のデータ構造が相互接続しやすい形になっているか
- 新規モジュール追加時に `main.py` と navigation の改修コストが過大にならないか
- `git` 更新後の Render 環境で、UI 崩れやルーティング不整合が起きないか
